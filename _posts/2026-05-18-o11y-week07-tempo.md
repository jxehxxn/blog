## 1. 비유 — "택배 추적 번호"

한 요청이 service A → B → C → D를 거침. 각 service에서 무엇을 했는지 + 얼마나 걸렸는지 추적 번호(trace_id) 하나로 묶어 보여주는 시스템.

## 2. 기본 개념

- **Trace**: 한 요청의 end-to-end. 여러 span의 트리.
- **Span**: 한 service 안의 한 작업 단위. start/end time, attribute.
- **Context propagation**: trace_id/span_id를 HTTP header로 전달 (W3C Trace Context 표준).

## 3. Tempo 아키텍처

```
[ App + OTel SDK ] → spans
       ↓ OTLP gRPC
[ Tempo distributor ]
       ↓
[ ingester ] → S3 (blocks)
       ↓
[ querier ] ← S3
       ↓
[ Grafana ]
```

Loki와 매우 유사. Grafana Labs 같은 패밀리.

## 4. 설치

```bash
helm install tempo grafana/tempo -n monitoring
```

production: tempo-distributed chart로 micro service.

## 5. OpenTelemetry로 instrument

```go
import (
  "go.opentelemetry.io/otel"
  "go.opentelemetry.io/otel/sdk/trace"
)

func main() {
  provider := trace.NewTracerProvider(
    trace.WithBatcher(otlpExporter),
  )
  otel.SetTracerProvider(provider)

  tracer := otel.Tracer("myapp")
  ctx, span := tracer.Start(ctx, "handleRequest")
  defer span.End()
  // 자식 span
  childCtx, child := tracer.Start(ctx, "queryDB")
  defer child.End()
  ...
}
```

OTel은 8주차 깊게.

## 6. Sampling 전략

모든 trace 저장은 비현실적 (10k RPS × span). sampling.

### Head sampling
요청 시작 시점에 결정 (e.g., 1%만). 단순하지만 정보 손실.

### Tail sampling
trace 완료 후 결정. 흥미로운 trace (error, slow)만 보관. 정확하지만 collector 자원 필요.

OTel collector tail-sampling processor가 표준.

자세한 sampling은 [보충 4: Sampling].

## 7. Trace ID Correlation

trace_id를 logs/metrics에 포함:
- 로그: `trace_id=abc123 ...`
- 메트릭 exemplar: histogram의 특정 bucket이 어떤 trace에서 왔는지.

Grafana가 metric spike → 클릭 → 해당 trace → 클릭 → 해당 log 자동 탐색. **3대 신호 통합 디버깅.**

## 8. Service Map

trace 데이터로 service 간 호출 그래프 자동 생성. "service A가 B/C를 호출, p95 latency 100ms" 등.

빅테크 디버깅 도구의 핵심.

## 9. 운영 함정 5선

1. 100% sampling → 비용 폭증.
2. tail sampling collector OOM.
3. trace_id propagation 누락 → trace 끊김.
4. retention 너무 길게.
5. log/metric correlation 미설정.
