빅테크 기업에서 보안 운영을 하다 보면 가장 먼저 마주하는 벽은 기술이 아니라 비용이다. 하루에 수십 테라바이트의 로그가 쏟아지는 환경에서 모든 데이터를 스플렁크(Splunk)에 인덱싱하는 건 자살 행위나 다름없다. 라이선스 비용이 보안 예산을 전부 잡아먹기 때문이다. 13주 차 보너스 모듈에서는 빅테크 SIEM 아키텍트들이 어떻게 데이터를 걸러내고 비용을 아끼는지 실무적인 관점에서 다뤄보겠다.

### 1. 데이터 라우팅: 입구에서 결정하라

모든 로그가 스플렁크로 바로 들어오게 두면 안 된다. 중간에 완충 지대와 필터가 필요하다. 요즘 실무에서는 크리블 스트림(Cribl Stream)이나 아파치 카프카(Apache Kafka)를 버퍼이자 라우터로 쓴다.

카프카는 대량의 데이터를 안정적으로 받아주는 역할을 한다. 크리블은 그 위에서 데이터를 가공한다. 보안상 중요하지 않은 디버그 로그는 여기서 바로 버린다. 개인정보(PII)는 마스킹 처리해서 인덱싱 비용을 줄이고 컴플라이언스도 챙긴다. 필요한 데이터만 골라서 스플렁크로 보내고, 나머지는 저렴한 S3 저장소로 돌리는 게 핵심이다.

### 2. 스플렁크 인게스트 액션(Ingest Actions)

스플렁크 자체 기능으로도 데이터를 거를 수 있다. 인게스트 액션을 쓰면 데이터를 인덱싱하기 전에 정규표현식으로 필터링하거나 변환할 수 있다. 특히 `nullQueue`를 사용해 불필요한 로그를 삭제하는 기술은 필수다.

#### 실전 예제: 특정 로그 버리기 (nullQueue)

윈도우 이벤트 로그 중에서 성공한 로그인(EventCode 4624)이 너무 많아 라이선스를 압박한다고 가정하자. 특정 계정의 로그인 로그만 버리고 싶을 때 다음과 같이 설정한다.

**props.conf**
```conf
[source::WinEventLog:Security]
TRANSFORMS-set_nullqueue = drop_noisy_logs
```

**transforms.conf**
```conf
[drop_noisy_logs]
REGEX = (?m)Account Name:\s+(SYSTEM|NETWORK SERVICE)
DEST_KEY = queue
FORMAT = nullQueue
```

이 설정은 `SYSTEM`이나 `NETWORK SERVICE` 계정의 로그인 로그를 인덱싱하지 않고 바로 버린다. 라이선스 사용량을 즉각적으로 줄여주는 마법 같은 설정이다.

#### 개인정보 마스킹 예제

이메일 주소 같은 민감 정보를 별표(*)로 가리고 싶을 때도 `transforms.conf`를 쓴다.

**transforms.conf**
```conf
[mask_email]
REGEX = (?i)^(.*)([a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,})(.*)$
FORMAT = $1#####@#####$3
DEST_KEY = _raw
```

### 3. 이벤트에서 메트릭(Metrics)으로의 전환

모든 데이터를 원본 로그 형태로 보관할 필요는 없다. CPU 사용량이나 네트워크 트래픽 같은 수치 데이터는 메트릭으로 변환하는 게 훨씬 경제적이다. 메트릭은 일반 이벤트보다 용량을 90% 이상 적게 차지한다.

`mstats` 명령어를 사용하면 메트릭 데이터를 초고속으로 조회할 수 있다. 대시보드 로딩 속도도 빨라지고 라이선스 비용도 획기적으로 줄어든다. 로그의 형태가 아니라 숫자의 흐름이 중요하다면 무조건 메트릭 전환을 고려해야 한다.

### 정답

1. **nullQueue**: 이 값을 지정하면 데이터가 인덱싱 엔진에 도달하기 전에 파기된다.
2. **Cribl Stream (또는 Apache Kafka)**: 데이터 파이프라인의 유연성을 높여주는 핵심 도구들이다.
3. **메트릭 (Metrics)**: `mstats`를 통해 조회하며 저장 공간과 라이선스 비용을 크게 절감한다.

빅테크의 보안은 단순히 해킹을 막는 게 아니다. 쏟아지는 데이터 속에서 가성비 좋은 정보를 찾아내는 싸움이다. 오늘 배운 라우팅 기술만 제대로 익혀도 여러분의 SIEM 운영 능력은 한 단계 올라갈 것이다.
