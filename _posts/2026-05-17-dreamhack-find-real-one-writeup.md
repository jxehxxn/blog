## 들어가며

이번 글에서는 Dreamhack 웹해킹 워게임 **Find Real One** (난이도 Bronze 4) 문제를 처음 접하는 정보보안 입문자의 시점에서 단계별로 풀어봅니다. 이 문제는 _코드 취약점 분석_ 문제가 아니라 **HTML/DOM의 미세한 차이를 식별하는 능력**을 묻는, "웹 인스펙션의 ABC"에 해당하는 입문 문제입니다.

문제 페이지의 텍스트 설명은 사실상 비어 있고, ZIP을 풀어보면 `Readme.txt` 한 줄만 들어 있습니다.

```
개발자 도구로 빠른 웹 검사!
Quick web inspection with developer tools!
```

요컨대 "브라우저 개발자 도구로 페이지를 잘 들여다보라"가 유일한 힌트입니다. 문제 제목 **Find Real One** 그대로, _가짜 99개 속에서 진짜 하나_ 를 골라내는 게 핵심입니다.

> Spoiler: 최종 페이로드는 한 줄짜리 `curl`로 끝나며, 그 안에서 우리가 다운로드할 PNG에 플래그가 적혀 있습니다. 다만 그 한 줄에 도달하기까지 _"무엇이 무엇과 다른가"_ 를 인지하는 사고 과정이 이 문제의 학습 포인트입니다.

## 1. 문제 파일 들여다보기

문제 페이지의 "문제 파일 받기"로 받은 ZIP을 풀어보면 의외로 단출합니다.

```
Find-Real-One/
└── Readme.txt   ← 위 두 줄짜리 힌트만 들어 있음
```

웹 백엔드 소스도, Docker도, 서버 주소도 없습니다. 챌린지 목록을 다시 확인해 보면 이 문제는 **`needs_vm: false`** 로 표시되어 있습니다. 즉,

- 서버를 직접 부팅해서 공격하는 문제가 아니다.
- 분석 대상은 **이 챌린지 페이지 자체**(혹은 그 페이지가 참조하는 리소스)다.

이 시점에서 가설을 하나 세웁니다: **플래그가 챌린지 페이지의 HTML/리소스 어딘가에 평문 또는 이미지 형태로 숨어 있다.**

## 2. 챌린지 페이지의 "문제 설명" 영역을 의심하기

브라우저에서 문제 페이지를 열어 설명 영역을 보면, 같은 이미지가 빽빽하게 격자로 나열되어 있습니다. 처음 보면 단순히 "같은 그림이 잔뜩 있네" 정도로 지나치기 쉽습니다.

여기서 한 가지 사실을 떠올려야 합니다. Dreamhack의 문제 설명은 Markdown → HTML로 렌더링되므로, **`<img>`의 `src`** 와 그것을 감싸는 **`<a>`의 `href`** 가 _반드시 같으리라는 보장이 없습니다_. 작성자가 의도적으로 같은 썸네일을 보여주면서 하이퍼링크는 서로 다른 곳을 가리키게 만들 수 있습니다.

브라우저 개발자 도구의 콘솔에서 페이지 안의 모든 링크와 이미지를 한 번에 추출해 봅니다.

```js
const desc = document.querySelector('#challenge-description .markdown-content');
const links = Array.from(desc.querySelectorAll('a')).map(a => a.href);
const imgs  = Array.from(desc.querySelectorAll('img')).map(i => i.src);
({
  totalLinks: links.length,
  totalImgs:  imgs.length,
  uniqueLinks: [...new Set(links)],
  uniqueImgs:  [...new Set(imgs)],
});
```

실행 결과는 (다듬어서) 다음과 같습니다.

```json
{
  "totalLinks": 100,
  "totalImgs":  100,
  "uniqueLinks": [
    "https://dreamhack-media.s3.amazonaws.com/attachments/753773efa1332db6c0039b2117f33a74b2566e47a995bc6293ec06dddbab35e3.png",
    "https://dreamhack-media.s3.amazonaws.com/attachments/4ecde3bf99e2bd81a9777797ded0186461c21492e729a2ba48c571fdf9b767bc.png"
  ],
  "uniqueImgs": [
    "https://dreamhack-media.s3.amazonaws.com/attachments/7e4f9f8cac9d44a8b4df1e85e28fefd61d90895194b9fe706ab68626afe2b31d.png"
  ]
}
```

세 가지 숫자가 결정적입니다.

- 화면에 보이는 **썸네일 이미지(`<img>`)는 100개 모두 동일한 URL**(`7e4f9f8c...`).
- 그런데 그걸 감싸는 **하이퍼링크(`<a href>`)는 100개 중 99개가 `753773ef...`** 한 URL.
- **단 1개만** `4ecde3bf...` 로 다른 URL을 가리킵니다.

문제 제목 _"Find Real One"_ 의 정체가 여기서 드러납니다. 표면적으로는 동일해 보이는 100개의 셀 중, **단 하나의 셀에 걸린 링크만이 다른 파일**을 가리키고 있는 것입니다.

> 비유하자면, _"같은 모양의 흰 봉투 100장이 책상에 깔려 있는데, 그중 한 봉투만 다른 우편번호의 주소가 적혀 있다"_ 와 같습니다. 봉투(`<img>`)는 모두 같지만, 그 봉투가 들어 있는 _도착지_(`<a href>`)는 한 곳만 다른 것이지요.

## 3. 왜 사람 눈으로는 못 찾을까

99/100과 1/100 차이가 화면에서 보이지 않는 이유는, 그 차이가 _시각적 요소가 아닌 메타데이터_ 에 있기 때문입니다.

- 사용자가 보는 것: `<img src="...7e4f9f8c.png">` — 100칸 모두 같은 그림.
- 사용자에게 보이지 않는 것: `<a href="...">` 의 목적지.

링크 텍스트도, 이미지도 모두 동일하니 마우스를 천천히 올려보지 않는 한 _"다른 한 칸이 있다"_ 는 사실을 눈으로 절대 찾을 수 없습니다. 그래서 출제자는 친절하게도 `Readme.txt`에 **"개발자 도구"** 를 쓰라고 명시한 것이고, 우리는 콘솔 한 줄로 즉시 그 한 칸을 찾아냈습니다.

이건 실무에서도 같은 결의 위험을 갖습니다. 피싱 메일이 가장 즐겨 쓰는 패턴이 정확히 이것입니다.

```html
<a href="https://attacker.example/login">https://www.bank.com/login</a>
```

링크 _텍스트_ 는 정상 도메인을 보여주지만 실제 `href`는 공격자 사이트로 향합니다. 사람 눈은 보이는 텍스트만 믿지만, 브라우저는 `href`로 이동합니다. **눈으로 보이는 것과 실제로 동작하는 것이 다를 수 있다** — 이것이 모든 _Find Real One_ 류 인스펙션 문제의 본질입니다.

## 4. 진짜 하나를 받아서 열어보기

후보가 좁혀졌으니, 다른 URL 하나를 직접 받아 봅니다. 브라우저로 열어도 되지만, 자동화/재현성을 위해 `curl`을 쓰는 게 깔끔합니다.

```bash
curl -s -o real_one.png \
  "https://dreamhack-media.s3.amazonaws.com/attachments/4ecde3bf99e2bd81a9777797ded0186461c21492e729a2ba48c571fdf9b767bc.png"

file real_one.png
# real_one.png: PNG image data, 722 x 143, 8-bit/color RGBA, non-interlaced
```

PNG 파일로 정상 다운로드됩니다. 그림을 열어보면 검은 글씨로 단 한 줄, 플래그가 적혀 있습니다.

```
DH{Correct_Take_Flag_UwU}
```

이 값을 챌린지 페이지 하단의 `Flag 입력` 폼에 그대로 붙여 넣고 _제출하기_ 를 누르면 정답으로 인정됩니다. API 응답으로도 `is_completed: true`가 돌아오며, 문제 해결이 끝납니다.

## 5. 취약점 관점에서 한 단계 더 — 이 문제가 알려주는 것

이 챌린지는 "코드 버그를 찾는다"기보다 **공격자의 관찰 능력을 훈련**시킵니다. 그래도 정보보안적으로 의미 있는 두 가지 교훈이 있습니다.

### 5-1. 출력의 "보이는 부분"만 검증하는 것은 위험하다

페이지를 _렌더링 결과_ 만 본 사용자는 100개의 같은 이미지가 있다고 생각하지만, 실제 DOM은 다른 정보를 담고 있습니다. 마찬가지로 서비스에서

- 로그/감사 시스템이 "사용자에게 보이는 텍스트"만 기록하고 실제 `href`나 `action` 같은 _머신이 따라가는 값_ 을 누락한다면,
- 또는 관리자 페이지에서 첨부파일 목록을 썸네일로만 보여주고 실제 다운로드 URL을 따로 확인할 수 없다면,

공격자는 _보이는 부분_ 과 _동작하는 부분_ 의 분리만으로 충분한 우회 표면을 얻습니다. 이 차이가 실제로 잘 알려진 공격 클래스(예: 피싱, 클릭재킹, link manipulation, open redirect)의 토대가 됩니다.

### 5-2. _HTML/DOM_ 은 자료구조다, 사람 눈이 아닌 코드로 봐라

99/100 차이는 사람 눈으로 보면 _존재하지 않는 것과 같지만_, 코드로 보면 `[...new Set(links)].length === 2` 라는 즉시 확인 가능한 사실입니다. 브라우저 개발자 도구는 _보안 분석의 패시브 도구_ 로서, 다음과 같은 패턴이 익숙해질수록 풀이가 빨라집니다.

```js
// 다른 점이 있는가? — 모든 a[href] 의 종류 세기
new Set([...document.querySelectorAll('a')].map(a => a.href)).size;

// 숨겨진 입력은? — type="hidden" 만 추출
[...document.querySelectorAll('input[type=hidden]')]
  .map(i => [i.name, i.value]);

// 주석에 단서가? — HTML 주석 모두 추출
document.documentElement.outerHTML
  .match(/<!--[\s\S]*?-->/g) || [];
```

훨씬 어려운 문제들도 결국 _첫 30초_ 에 이 패턴들을 자동으로 돌려보는 습관에서 차이가 갈립니다.

## 6. 전체 풀이 흐름 한 줄 요약

1. 문제 파일 = `Readme.txt` 한 줄. → "개발자 도구로 보라".
2. 페이지 설명 영역에 100개의 동일한 이미지가 그리드로 깔려 있다.
3. 콘솔에서 `<a href>`를 모두 추출 → **고유값이 2개**, 99 vs 1.
4. 1개짜리 URL을 `curl`로 받음 → PNG, 그림 안에 **`DH{Correct_Take_Flag_UwU}`**.
5. 플래그 제출, 끝.

## 7. 정리 — 입문자가 가져갈 교훈

- **눈에 보이는 것** 과 **DOM에 적힌 것** 은 다를 수 있다. 보안 분석은 항상 후자를 본다.
- 브라우저 개발자 도구의 콘솔은 "보안용 grep"이다. `querySelectorAll → Set → length` 패턴 하나로 _이상치(outlier)_ 가 즉시 보인다.
- 챌린지 설명에 _리소스(이미지/링크)_ 가 등장하면, 그 리소스의 URL과 파일 자체를 모두 의심해야 한다. 메타데이터(EXIF), 파일 내부 텍스트, 다중 첨부 중 _하나만 다른 것_, ZIP/PDF 안의 추가 파일 등 _숨길 곳_ 은 많다.
- "no VM" 표시는 종종 "분석 대상은 _문제 페이지 그 자체_"라는 신호다.

이 문제 풀이 자체는 1분이면 충분하지만, 거기서 끄집어낼 수 있는 습관(**보이는 것을 믿지 말고 DOM을 읽어라**)은 앞으로 모든 웹 워게임에서 첫 30초의 가치를 결정합니다.

## 부록 — 시도하다 막혔던 지점과 회고

이번 풀이는 운 좋게 빠르게 끝났지만, _원래 풀려고 했던 다른 문제_ 에서 막혀 우회를 한 과정이 있었습니다. 같은 함정을 다시 밟지 않기 위해 기록합니다.

### 실패 1. 처음 고른 문제는 VM이 필요했는데, VM 크레딧이 0이었다

원래는 **Switching Command** (id=1081, Silver 4) 를 풀려고 했습니다. PHP 8 + MariaDB 구성에 두 단계 취약점이 있는 좋은 문제(_PHP `switch` 의 느슨한 비교 우회_ + _`escapeshellcmd` 의 인용부호 페어 통과를 악용한 `\bflag\b` 워드 바운더리 우회_)였고, 분석까지는 다 끝낸 상태였습니다. 그러나 서버 생성을 누르려는 순간 _"VM 크레딧 0, 충전까지 6시간 59분"_ 메시지가 떴습니다. VM 크레딧은 매주 월요일에 충전되며, 오늘은 일요일이라 사실상 그날 안에는 해결이 불가능했습니다.

**회고**: 분석 시작 _전_ 에 반드시 두 가지를 확인한다.

1. 해당 챌린지가 `needs_vm: true` 인지 (`/api/v1/wargame/challenges/{id}/` 응답에 들어 있다).
2. `needs_vm: true` 라면, _현재 보유 VM 크레딧_ 이 0이 아닌지 (`/wargame/challenges/{id}` 페이지의 `.challenge-live` 영역에 명시된다).

다음부터는 문제를 선정하는 첫 단계에서 _"VM이 필요한가 → 크레딧이 있는가"_ 를 게이트로 두고, 둘 다 통과한 경우에만 소스 코드 분석에 들어가겠습니다. 분석을 다 끝낸 뒤 _공격 단계에 못 들어가는 것_ 은 가장 큰 시간 낭비입니다.

### 실패 2. `category=web, status=todo` 첫 페이지에서 `needs_vm: false` 인 문제는 단 두 개뿐이었다

VM이 필요 없는 후보를 좁히려고 검색을 다시 했더니, 풀이자 많은 순 첫 200문제 중 _no-VM_ + _미해결_ 만 추리면 단 두 개였습니다(`id=2182`, `id=2188`). 또한 `id=2182` 는 _제목 자체가 보이지 않는 유니코드 문자_ 라 페이지에 따로 단서가 없어 보였고, 결국 _Find Real One_ 으로 결정했습니다.

**회고**: Dreamhack은 _대부분의 웹 문제가 VM 기반_ 이라는 사실을 기본 가정으로 둔다. 크레딧이 떨어진 상황에서는 _문제 풀이 활동 자체가 사실상 정지_ 되므로, 평소에 _no-VM_ 문제를 따로 모아 둔 _백업 리스트_ 를 유지해두면 다음에 같은 상황에서 즉시 풀이 가능한 문제를 고를 수 있다.

### 실패 3. CSRF 토큰을 직접 헤더에 넣어 live API를 호출해 보았으나 거부됨

VM 크레딧이 정말 막혔는지 확인하려고 `POST /api/v1/wargame/challenges/1081/live/` 를 콘솔에서 직접 호출해 봤지만 `"CSRF Failed: CSRF token from the 'X-Csrftoken' HTTP header has incorrect length."` 가 돌아왔습니다. Dreamhack은 _쿠키의 csrftoken 값을 그대로 X-CSRFToken 헤더에 복사_ 하는 기본 패턴 대신, 추가적인 토큰 검증을 두고 있는 것으로 보입니다.

**회고**: 결국 _UI 버튼을 통한 정상 흐름_ 으로만 VM 생성이 가능하며, UI에서 버튼이 비활성화되어 있다면 우회할 방법이 없다. 따라서 _VM이 필요한 문제_ 와 _크레딧 부족_ 의 조합은 곧 _다음 주 월요일까지 대기_ 와 동의어다.

이 세 가지 실패에서 가장 큰 교훈은 _"풀이 방향을 정하기 전에 인프라 가용성을 먼저 확인하라"_ 입니다. 코드 분석 능력이 아무리 좋아도 공격할 대상이 살아 있지 않다면 풀이는 시작도 안 됩니다.
