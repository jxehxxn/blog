## 1. 시크릿이란 무엇인가 — "비밀"보다 더 정확한 정의

비유로 시작합시다. 시크릿은 **호텔 마스터키**와 같습니다. 일반 직원이 객실 키만 받는 것과 달리, 마스터키는 모든 객실을 엽니다. 마스터키가 길거리에 떨어지면 단순히 "분실"이 아니라 **호텔 전체의 보안 모델이 붕괴**됩니다.

기술적으로 시크릿은 다음 속성을 모두 갖는 데이터입니다.

1. **인증/인가 권한을 부여한다** (단순 식별자 ≠ 시크릿)
2. **장기간(>1분) 유효하다** (1회용 OTP는 보통 시크릿 분류에서 제외)
3. **유출 시 즉시 회수(revoke) 가능해야 한다** (회수 불가능한 것은 시크릿이 아니라 "신원" — 예: 개인 SSN)

### 시크릿 분류

| 분류 | 예시 | 회수 난이도 |
|------|------|-------------|
| API 토큰 | AWS Access Key, GitHub PAT, Stripe sk_live | 쉬움 (콘솔에서 revoke) |
| OAuth 클라이언트 시크릿 | Google OAuth, Slack App | 중간 (앱 재발급 필요) |
| 개인키 (asymmetric) | SSH private key, JWT signing key | 어려움 (키 페어 전체 교체) |
| 패스워드 | DB 비밀번호, root 패스워드 | 중간 (변경 후 클라이언트 모두 재배포) |
| 세션 토큰 | 쿠키, JWT | 쉬움 (서버에서 invalidate) |
| 인프라 자격 증명 | kubeconfig, .pem | 어려움 (인증 체인 전체 영향) |

빅테크 환경에서 가장 위험한 건 **장기 유효 + 광범위 권한 + 회수 어려움**의 교집합입니다. 대표적으로 **AWS Root Access Key**, **GitHub Org Admin PAT**, **DB SuperUser 패스워드**가 여기 해당합니다.

## 2. 실제 사례 3선 — 회사를 흔든 시크릿 누출

### 사례 1: Uber 2016 — GitHub Private Repo의 AWS 키

2016년, Uber 엔지니어가 사내 GitHub Private 리포지토리에 AWS 키를 커밋했습니다. 공격자가 약한 패스워드로 한 엔지니어 계정을 인수(Account Takeover)하자 즉시 AWS 인프라로 횡적 이동(lateral movement)했고, 5,700만 명의 운전자/승객 데이터가 유출됐습니다. **Uber는 FTC와 1억 4,800만 달러 합의금**을 지불했고, 당시 CSO는 형사 기소까지 됐습니다.

교훈: **Private Repo도 안전지대가 아니다.** 시크릿은 코드와 분리해 secret manager에 두어야 합니다.

### 사례 2: Toyota 2022 — 5년간 노출된 액세스 토큰

2022년 10월 Toyota는 공식 GitHub에 5년 전 커밋된 액세스 토큰이 외부에 노출되어 약 30만 고객의 정보가 위험에 노출됐다고 발표했습니다. 토큰은 데이터베이스 서버 접근용이었고, 5년간 아무도 발견하지 못했습니다.

교훈: **이미 푸시된 시크릿은 `git rm`으로 지워지지 않는다.** Git 히스토리, fork, 미러, 캐시 어딘가에 남아 있습니다. TruffleHog가 4주차에서 다룰 영역입니다.

### 사례 3: Twitch 2021 — 125 GB 소스코드 + 시크릿 일괄 노출

해커가 Twitch 서버 설정 오류를 통해 125 GB의 소스코드, 스트리머 수익 데이터, 미공개 게임 클라이언트, 그리고 다수의 API 키·내부 인증 정보를 일괄 노출했습니다. 노출된 시크릿은 수백 개로 추정되며, Amazon은 Twitch 전체 인프라 자격 증명 회전(rotation) 작업에 며칠을 소요했습니다.

교훈: **시크릿 회전이 사고 발생 후가 아니라 평소에 연습되어 있어야 한다.** 10주차 인시던트 대응에서 자세히 다룹니다.

## 3. TruffleHog가 경쟁 도구와 다른 점

같은 카테고리에 Gitleaks, detect-secrets, GitGuardian이 있습니다. 한눈에 비교하면:

| 항목 | TruffleHog v3 | Gitleaks | detect-secrets | GitGuardian |
|------|--------------|----------|----------------|-------------|
| 언어 | Go | Go | Python | SaaS |
| 라이브 검증(verifier) | **있음 (700+ detector)** | 없음 | 없음 | 있음 (SaaS) |
| 셀프호스트 무료 | O | O | O | X (Enterprise만) |
| Org 단위 스캔 | O | 제한적 | X | O |
| Docker/S3/Postman 스캔 | **O** | X | X | O |
| 커스텀 디텍터 | regex + verifier 모두 | regex만 | regex만 | regex만 |

핵심 차별점은 **라이브 검증(verifier)**입니다. TruffleHog는 AWS 키처럼 보이는 문자열을 발견하면 STS `GetCallerIdentity`를 호출해 **실제로 살아있는 키인지 확인**합니다. 이것이 빅테크 환경에서 결정적인 이유: 매일 수천 건의 의심 패턴이 나오는데 **verified만 골라 대응**하면 인시던트 큐가 폭발하지 않습니다.

## 6. 자가평가 퀴즈

### Q1. 다음 중 "시크릿"이 아닌 것은?
1. AWS Access Key
2. JWT signing private key
3. 사용자 이메일 주소
4. DB 슈퍼유저 패스워드

**정답:** 3번. 이메일 주소는 식별자(identifier)일 뿐 그 자체로 권한을 부여하지 않습니다. 다만 피싱·계정탈취의 진입점이 되므로 PII로는 보호 대상입니다.

### Q2. TruffleHog v3의 "verifier"가 하는 일은?
1. 정규표현식 매칭의 정확도를 통계로 계산한다.
2. 발견된 자격 증명을 실제 서비스에 호출해 살아있는지 확인한다.
3. Git 히스토리를 압축한다.
4. False Positive를 자동으로 베이스라인에 추가한다.

**정답:** 2번. Verifier는 detector가 매칭한 문자열을 실제 API에 시험 호출(예: AWS STS GetCallerIdentity, GitHub `/user`)해 200 응답이 오면 verified로 분류합니다. 3주차와 보충 2에서 깊게 다룹니다.

### Q3. Uber 2016 사고에서 가장 큰 교훈은?
1. Public 리포지토리만 위험하다.
2. Private 리포지토리도 계정 탈취 한 번이면 안전지대가 아니다.
3. AWS는 root 키를 절대 만들지 말아야 한다.
4. GitHub PAT는 무한 유효 기간이어야 한다.

**정답:** 2번. Private 리포지토리는 코드 노출은 막아주지만, 계정이 탈취되면 그 안의 시크릿이 그대로 노출됩니다. "방어 심층화(defense in depth)"의 필요성을 보여줍니다.

### Q4. 빅테크 환경에서 시크릿 회수가 가장 어려운 것은?
1. 24시간 만료되는 OAuth 액세스 토큰
2. 비대칭 키 페어의 개인키 (예: JWT signing key)
3. 1회용 OTP
4. 세션 쿠키

**정답:** 2번. 개인키가 누출되면 그 키로 서명된 모든 토큰을 신뢰할 수 없게 되어, **키 페어 전체 교체 + 발급된 토큰 전부 무효화 + 모든 검증 클라이언트 재배포**가 필요합니다.

### Q5. TruffleHog v3와 detect-secrets의 가장 큰 차이는?
1. 언어가 다르다.
2. TruffleHog는 라이브 검증을 지원하고 detect-secrets는 그렇지 않다.
3. detect-secrets는 GitHub Org 스캔을 지원한다.
4. 둘은 사실상 동일한 도구다.

**정답:** 2번. detect-secrets는 entropy + regex 기반의 정적 탐지에 머무릅니다. TruffleHog는 700+ 디텍터에 라이브 검증기를 결합해 "verified" 신호를 만들어냅니다.
