## 학습 목표

- SBOM이란 + 왜 필요.
- 2가지 표준 (SPDX, CycloneDX).
- Syft로 자동 생성.
- SBOM 보관 + 검색.

## 1. 비유 — "식품 성분표"

빅맥에는 빵·고기·소스가 들어 있고, 빵에는 밀가루·물·소금이 들어 있고... 모든 성분을 표시한 라벨이 SBOM.

소프트웨어는: 내 앱이 어떤 lib 어떤 버전을 쓰는지. transitive 까지.

## 2. SBOM 종류

- **SPDX** (Linux Foundation): 가장 오래됨, ISO 표준.
- **CycloneDX** (OWASP): vulnerability 중심.

빅테크 둘 다 지원. 변환 도구 풍부.

## 3. Syft

```bash
brew install syft
syft mycorp/myapp:v1.2.3 -o spdx-json > sbom.json
syft mycorp/myapp:v1.2.3 -o cyclonedx-json > sbom.cdx.json
```

container image 또는 디렉토리 input.

## 4. SBOM 구조

```json
{
  "spdxVersion": "SPDX-2.3",
  "packages": [
    {
      "name": "log4j-core",
      "version": "2.14.1",
      "supplier": "Organization: Apache",
      "downloadLocation": "...",
      "filesAnalyzed": true,
      "checksums": [{"algorithm": "SHA256", "checksumValue": "..."}],
      "licenseConcluded": "Apache-2.0",
      "externalRefs": [
        {"referenceType": "purl", "referenceLocator": "pkg:maven/org.apache.logging.log4j/log4j-core@2.14.1"}
      ]
    },
    ...
  ]
}
```

## 5. 보관

OCI image attestation으로 push:
```bash
cosign attest --predicate sbom.json --type spdxjson mycorp/myapp:v1.2.3
```

또는 별도 SBOM database (Anchore, OWASP DT).

## 6. 검색 + 영향 분석

Log4Shell 같은 사고 시 "log4j-core 2.14.x 사용 image 모두" 검색.

```bash
# Anchore: 
anchorectl source content / list --type spdx
```

이게 SBOM의 진짜 가치.

## 7. Dependency Track (OWASP)

자체 호스팅 SBOM database + vuln tracking.

```bash
docker run -d -p 8080:8080 dependencytrack/bundled
```

신규 CVE 발견 시 영향받은 모든 project 자동 알람.

## 8. Generation Layer

- 소스 layer: source tree analysis.
- Build layer: build system이 생성 (Bazel, Maven).
- Container layer: image scan (Syft).
- Runtime layer: 실행 중 동적 발견 (덜 표준).

빅테크 표준: build + container 둘 다.

## 9. 운영 함정

1. SBOM 생성만 + 저장/검색 없음 → 의미 없음.
2. 변환 손실 (SPDX ↔ CDX 일부 정보 손실).
3. License 정확성 부족.
4. private lib 정보 누출 (외부 SBOM 공개 시).
5. transitive 깊이 미보고.
