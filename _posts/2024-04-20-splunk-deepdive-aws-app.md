# Chapter 1: Architecting Visibility in the AWS Cloud

In the modern enterprise, visibility is not just a luxury; it's a survival requirement. As organizations migrate mission-critical workloads to Amazon Web Services (AWS), the complexity of the infrastructure grows exponentially. Traditional monitoring tools often fail to keep pace with the ephemeral nature of cloud resources. This chapter provides a deep dive into the Splunk App for AWS, focusing on the architectural decisions, data collection mechanisms, and security best practices required to build a world-class security operations center (SOC) in the cloud.

---

## 1. The AWS Logging Ecosystem: A Deep Dive

Before we can analyze data in Splunk, we must understand how that data is generated and captured within AWS. There are three primary pillars of AWS logging that every security professional must master.

### 1.1 AWS CloudTrail: The Audit Trail of the Cloud
CloudTrail is the foundation of AWS security. It records every API call made in your account, whether through the AWS Management Console, SDKs, or CLI.

*   **Management Events**: These provide insights into control plane operations, such as creating an EC2 instance or deleting an S3 bucket.
*   **Data Events**: These provide insights into resource-level operations, such as S3 object-level actions (Get, Put, Delete) or Lambda function executions. These are high-volume and can significantly impact costs.
*   **Insights Events**: These use machine learning to detect unusual API activity, such as a sudden spike in `TerminateInstances` calls.

**Generation & Collection**:
CloudTrail events are generated by the AWS control plane. By default, they are stored in the CloudTrail service for 90 days. For long-term analysis in Splunk, you must configure a "Trail" to deliver these logs to an S3 bucket.

### 1.2 Amazon CloudWatch Logs: The Application Pulse
CloudWatch Logs is the central repository for logs from various AWS services and your own applications.

*   **Vended Logs**: Logs generated by AWS services on your behalf (e.g., VPC Flow Logs, Route 53 logs).
*   **Custom Logs**: Logs sent from your EC2 instances via the CloudWatch Logs Agent or from Lambda functions.

**Generation & Collection**:
Logs are organized into **Log Groups** and **Log Streams**. To get these into Splunk, you typically use a CloudWatch Logs Subscription Filter to push data to a Kinesis Data Firehose.

### 1.3 VPC Flow Logs: The Network Map
VPC Flow Logs capture information about the IP traffic going to and from network interfaces in your VPC.

*   **Metadata Captured**: Source/Destination IP, Source/Destination Port, Protocol, Packets, Bytes, and Action (ACCEPT/REJECT).
*   **Capture Window**: Data is aggregated over a 1-minute or 10-minute interval.

**Generation & Collection**:
Flow logs can be published to S3 or CloudWatch Logs. For Splunk integration, publishing to S3 and using SQS-based collection or pushing to Firehose are the preferred methods.

## 2. Data Collection Architectures: Pull vs. Push

Choosing the right architecture is critical for scalability and cost-efficiency.

### 2.1 The Pull Model: SQS-based S3
In this model, AWS services drop logs into an S3 bucket. An S3 Event Notification triggers a message to an SQS queue. The Splunk Add-on for AWS polls the SQS queue and downloads the logs from S3.

*   **Pros**: Simple to set up, cost-effective for low-to-medium volumes.
*   **Cons**: Latency (polling interval), scaling bottlenecks at very high volumes.

### 2.2 The Push Model: Kinesis Data Firehose to HEC (The Gold Standard)
For enterprise-scale environments, the push model is superior. AWS services push logs to Kinesis Data Firehose, which then streams them directly to the Splunk HTTP Event Collector (HEC).

*   **Pros**: Real-time delivery, massive scalability, no polling overhead.
*   **Cons**: Requires HEC setup and potential load balancer configuration.

## 3. Implementation Guide: Setting Up the Infrastructure

### 3.1 AWS CLI Setup Commands
Use the following commands to initialize your logging infrastructure.

```bash
# 1. Create an S3 bucket for CloudTrail logs
aws s3api create-bucket --bucket my-splunk-logging-bucket --region us-east-1

# 2. Create a CloudTrail trail
aws cloudtrail create-trail --name enterprise-audit-trail --s3-bucket-name my-splunk-logging-bucket

# 3. Start logging
aws cloudtrail start-logging --name enterprise-audit-trail

# 4. Create a Kinesis Data Firehose delivery stream for Splunk
aws firehose create-delivery-stream \
    --delivery-stream-name SplunkDeliveryStream \
    --delivery-stream-type DirectPut \
    --splunk-destination-configuration '{
        "HECEndpoint": "https://splunk-hec.example.com:8088",
        "HECToken": "your-hec-token-here",
        "HECEndpointType": "Raw",
        "S3Configuration": {
            "RoleARN": "arn:aws:iam::123456789012:role/FirehoseToS3Role",
            "BucketARN": "arn:aws:iam::123456789012:bucket/firehose-backup-bucket"
        }
    }'
```

### 3.2 IAM Policy JSONs: Least Privilege in Action
Security is paramount. Never use root or admin keys. Use dedicated IAM roles with specific policies.

**Policy for Splunk Add-on (SQS-based S3):**
```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-splunk-logging-bucket",
                "arn:aws:s3:::my-splunk-logging-bucket/*"
            ]
        },
        {
            "Effect": "Allow",
            "Action": [
                "sqs:ReceiveMessage",
                "sqs:DeleteMessage",
                "sqs:GetQueueAttributes"
            ],
            "Resource": "arn:aws:sqs:us-east-1:123456789012:splunk-s3-notifications"
        }
    ]
}
```

**Policy for Kinesis Firehose to Splunk:**
```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:AbortMultipartUpload",
                "s3:GetBucketLocation",
                "s3:GetObject",
                "s3:ListBucket",
                "s3:ListBucketMultipartUploads",
                "s3:PutObject"
            ],
            "Resource": [
                "arn:aws:s3:::firehose-backup-bucket",
                "arn:aws:s3:::firehose-backup-bucket/*"
            ]
        },
        {
            "Effect": "Allow",
            "Action": [
                "logs:PutLogEvents"
            ],
            "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/kinesisfirehose/SplunkDeliveryStream:*"
        }
    ]
}
```

## 4. Splunk Configuration: The Final Mile

Once the data is flowing, you must configure the Splunk App for AWS to interpret it correctly.

1.  **Install the Add-on**: The Splunk Add-on for AWS is responsible for data ingestion and CIM (Common Information Model) mapping.
2.  **Install the App**: The Splunk App for AWS provides the dashboards and visualizations.
3.  **Configure Macros**: Ensure the `aws-account-id` and `aws-index` macros are correctly set to point to your data.
4.  **HEC Tuning**: For high-volume Firehose streams, increase the HEC `max_content_length` and ensure your indexers are properly load-balanced.

---

## 5. Best Practices for the Big Tech Professional

*   **Filter at the Source**: Use VPC Flow Log filters (e.g., only REJECT traffic) to save on Splunk license costs.
*   **Use Cross-Account Roles**: If you have multiple AWS accounts, use a centralized logging account and grant Splunk access via Cross-Account IAM Roles.
*   **Monitor Firehose Health**: Set up CloudWatch Alarms for `DeliveryToSplunk.Success` and `DeliveryToS3.Success` metrics.
*   **Encryption**: Always use KMS for encrypting logs at rest in S3 and TLS for data in transit to Splunk HEC.

## Self-Assessment: Test Your Knowledge

1.  **Scenario**: You need to detect if an unauthorized user is trying to access sensitive objects in an S3 bucket. Which AWS log source and event type do you need?
2.  **Architecture**: Why is Kinesis Data Firehose preferred over SQS-based S3 for a global e-commerce platform during Black Friday?
3.  **Security**: What is the risk of using an IAM User Access Key instead of an IAM Role for Splunk authentication?

### Answers
1.  **CloudTrail Data Events** (specifically `GetObject` actions).
2.  **Scalability and Real-time Delivery**. Firehose can handle massive bursts of traffic without the latency associated with SQS polling.
3.  **Credential Exposure**. Access keys are static and can be leaked. IAM Roles use temporary security tokens and are much harder to compromise.
