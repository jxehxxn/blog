현대 클라우드 보안에서 가시성은 생명입니다. AWS 환경이 복잡해질수록 로그를 단순히 쌓는 것을 넘어 어떻게 효율적으로 분석하느냐가 관건이죠. 30년 동안 빅테크 현장에서 보안 인프라를 다루며 겪은 시행착오를 바탕으로 Splunk App for AWS의 핵심 설정과 아키텍처 선택의 이유를 정리했습니다.

### 1. 인증의 기초: IAM Role vs User Key
많은 입문자가 실수하는 지점이 바로 IAM User의 Access Key를 Splunk에 직접 입력하는 방식입니다. 보안 전문가라면 절대 권장하지 않습니다. 키 유출 시 피해가 막심하고 관리도 어렵기 때문이죠.

대신 IAM Role을 사용하세요. Splunk 인스턴스가 AWS 내부에 있다면 EC2 Role을 부여하고, 외부에 있다면 Cross-Account Role을 통해 임시 토큰을 발급받는 방식이 훨씬 안전합니다. 최소 권한 원칙(Least Privilege)에 따라 필요한 정책만 연결하는 습관을 들여야 합니다.

### 2. 데이터 수집 방식의 선택: SQS-based S3 vs Kinesis Firehose
로그를 가져오는 방법은 크게 두 가지입니다. S3에 저장된 로그를 SQS 알림을 통해 긁어오는 방식과, Kinesis Firehose를 통해 실시간으로 밀어넣는 방식이죠.

소규모 환경이나 과거 로그 분석이 주 목적이라면 SQS-based S3 방식이 무난합니다. 설정이 쉽고 비용이 저렴하니까요. 하지만 초당 수만 건의 로그가 발생하는 빅테크 환경에선 이야기가 달라집니다.

대규모 환경에선 Kinesis Data Firehose를 사용해 Splunk HEC(HTTP Event Collector)로 직접 데이터를 푸시하는 아키텍처를 선택해야 합니다. 이 방식은 확장성이 뛰어나고 지연 시간이 거의 없습니다. 중간에 S3를 거치지 않아도 되니 관리 포인트도 줄어들죠.

### 3. 주요 로그 온보딩 가이드
보안 가시성을 위해 반드시 챙겨야 할 세 가지 로그가 있습니다.

첫째는 CloudTrail입니다. 누가, 언제, 어떤 API를 호출했는지 기록하는 감사 로그의 핵심이죠. 모든 리전의 로그를 하나의 S3 버킷으로 모으는 것이 관리 효율 면에서 유리합니다.

둘째는 VPC Flow Logs입니다. 네트워크 트래픽의 흐름을 파악해 이상 징후를 탐지하는 데 필수적입니다. 양이 굉장히 많으므로 필요한 서브넷이나 인터페이스만 골라 수집하는 전략이 필요할 때도 있습니다.

셋째는 GuardDuty입니다. AWS의 지능형 위협 탐지 서비스로, 발견된 위협(Findings)을 Splunk로 가져와 대시보드에서 한눈에 확인해야 합니다.

### 4. 빅테크의 선택: Firehose to HEC 아키텍처
글로벌 서비스 규모에선 데이터 처리량이 상상을 초월합니다. 이때 Firehose와 Splunk HEC 조합은 빛을 발하죠. Firehose가 데이터를 배치로 묶어 HEC로 쏘아주면 Splunk는 이를 병렬로 처리합니다.

부하 분산(Load Balancing)을 위해 HEC 앞에 로드 밸런서를 두는 것도 잊지 마세요. 이렇게 구성하면 특정 인덱서에 장애가 생겨도 데이터 유실 없이 안정적인 수집이 가능해집니다.
