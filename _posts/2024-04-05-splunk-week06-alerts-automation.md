보안 관제 센터(SOC)에서 가장 무서운 적은 해커가 아닙니다. 바로 '알람 피로(Alert Fatigue)'입니다. 하루에 수천 개씩 쏟아지는 알람 속에서 진짜 위협을 찾아내는 일은 모래사장에서 바늘 찾기보다 어렵습니다. 6주차인 이번 시간에는 단순한 검색을 넘어, 어떻게 하면 분석가를 지치지 않게 하면서도 정확한 탐지를 수행할 수 있는지 알아보겠습니다.

### 1. 알람 설정의 기본: 검색에서 액션으로

Splunk에서 알람을 만드는 과정은 간단합니다. 잘 짜인 SPL 검색 결과를 저장할 때 'Alert' 타입을 선택하면 됩니다. 하지만 실무에서는 여기서부터 고민이 시작됩니다.

알람은 크게 두 가지 방식으로 작동합니다.
* **실시간(Real-time):** 조건이 충족되는 즉시 발생합니다. CPU 사용량이 99%를 찍는 순간 바로 알아야 하는 상황에 적합합니다.
* **스케줄(Scheduled):** 5분마다 혹은 1시간마다 정해진 주기에 따라 검색을 수행합니다. 대부분의 보안 탐지는 시스템 부하를 줄이기 위해 이 방식을 권장합니다.

### 2. 임계값(Threshold)과 억제(Throttling)

단순히 "로그가 찍히면 알려줘"라고 설정하면 지옥을 맛보게 됩니다. 특정 IP에서 로그인 실패가 1번 발생했다고 알람을 띄우면, 비밀번호를 한 번 틀린 모든 직원이 보안 위협으로 간주됩니다.

이때 필요한 것이 임계값입니다. "5분 이내에 동일한 IP에서 로그인 실패가 10회 이상 발생할 때"처럼 조건을 구체화해야 합니다.

더 중요한 기술은 '억제(Throttling)'입니다. 한 번 알람이 발생했다면, 동일한 이슈에 대해서는 일정 시간 동안 추가 알람을 보내지 않도록 설정하는 기능입니다. 예를 들어 서버 한 대가 죽어서 1초에 100개의 에러 로그를 뿜어낼 때, 휴대폰으로 100개의 문자가 오게 해서는 안 됩니다. 첫 알람 이후 30분간은 조용히 하도록 설정하는 것이 분석가의 정신 건강에 이롭습니다.

### 3. 오탐(False Positive) 줄이기: 분석가의 숙명

보안 장비가 뱉어내는 알람의 90% 이상은 실제 위협이 아닌 오탐인 경우가 많습니다. 오탐을 줄이는 작업은 탐지 규칙을 만드는 것보다 훨씬 중요합니다.

가장 먼저 해야 할 일은 '화이트리스트' 관리입니다. 취약점 점검 도구가 돌고 있거나, 정기적인 백업 작업 때문에 발생하는 대량의 트래픽은 탐지 제외 대상에 넣어야 합니다. 

또한, 컨텍스트를 추가하십시오. 단순히 "비정상적인 프로세스 실행"을 잡는 게 아니라, "평소에 이 사용자가 쓰지 않던 프로세스가 실행됨"과 같이 사용자 행동 패턴을 결합하면 정확도가 비약적으로 올라갑니다.

### 4. 외부 연동: Slack과 Jira 활용

Splunk 안에서만 알람을 보는 시대는 지났습니다. 팀원들이 가장 많이 머무는 곳으로 정보를 보내야 합니다. Webhook을 이용하면 Slack 채널에 메시지를 뿌리거나, Jira에 자동으로 티켓을 생성할 수 있습니다.

아래는 Slack으로 보낼 Webhook 페이로드의 예시입니다. Python을 이용해 커스텀 알람 액션을 만들 때 참고할 수 있는 구조입니다.

```python
import json
import requests

def send_slack_alert(webhook_url, alert_data):
    # Slack 메시지 포맷 구성
    payload = {
        "text": "🚨 Splunk 보안 탐지 알람 발생!",
        "attachments": [
            {
                "color": "#eb4034",
                "fields": [
                    {"title": "알람 명칭", "value": alert_data.get("search_name"), "short": True},
                    {"title": "위험도", "value": "High", "short": True},
                    {"title": "요약", "value": f"사용자 {alert_data.get('user')}의 비정상 접근 감지", "short": False}
                ]
            }
        ]
    }
    
    response = requests.post(
        webhook_url, 
        data=json.dumps(payload),
        headers={'Content-Type': 'application/json'}
    )
    
    if response.status_code != 200:
        print(f"전송 실패: {response.text}")

# 예시 데이터
example_data = {
    "search_name": "Brute Force Attack Detected",
    "user": "admin_test"
}
# send_slack_alert("https://hooks.slack.com/services/...", example_data)
```

### 5. SOAR로 가는 길: 자동화의 완성

알람이 발생한 뒤에 분석가가 수동으로 IP를 조회하고, 평판 사이트를 뒤지고, 방화벽 차단 정책을 넣는 과정은 너무 느립니다. 여기서 SOAR(Security Orchestration, Automation and Response)가 등장합니다.

Splunk SOAR(구 Phantom)는 Splunk에서 발생한 알람을 넘겨받아 미리 정의된 '플레이북(Playbook)'을 실행합니다.
1. Splunk에서 알람 발생
2. SOAR가 해당 IP의 평판을 VirusTotal에서 자동 조회
3. 위험 점수가 높다면 Slack으로 분석가에게 승인 요청 메시지 전송
4. 분석가가 '승인' 버튼을 누르면 방화벽에서 해당 IP 자동 차단

이런 일련의 과정을 자동화하면 분석가는 단순 반복 작업에서 벗어나 더 고도화된 위협 사냥(Threat Hunting)에 집중할 수 있습니다.

### 6주차 자가 진단 퀴즈

**문제 1:** 동일한 조건의 알람이 너무 자주 발생하여 분석가가 피로를 느낄 때, Splunk 알람 설정에서 가장 먼저 검토해야 하는 기능은 무엇인가요?
**문제 2:** 실시간(Real-time) 알람보다 스케줄(Scheduled) 알람을 권장하는 주된 이유는 무엇인가요?
**문제 3:** SOAR의 핵심 기능 중 하나로, 여러 보안 도구를 하나로 묶어 자동화된 절차를 수행하는 시나리오를 무엇이라 부르나요?

---

### 정답 및 해설

**정답 1:** 억제(Throttling). 특정 시간 동안 중복 알람이 나가지 않도록 제어해야 합니다.
**정답 2:** 시스템 자원 관리. 실시간 검색은 인덱서에 지속적인 부하를 주지만, 스케줄 검색은 정해진 시간에만 자원을 사용하므로 훨씬 안정적입니다.
**정답 3:** 플레이북(Playbook). 보안 사고 대응 절차를 코드로 구현한 자동화 시나리오입니다.
