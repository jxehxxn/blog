12주를 마친 학습자가 본인 수준을 가늠할 종합 퀴즈입니다. 모든 문항은 4지선다, 정답 + 해설을 포함합니다.

## 섹션 A — 기초 (Q1~Q10)

### Q1. "시크릿"의 핵심 속성 3가지에 가장 가까운 것은?
1. 길이, 색상, 위치
2. 인증/인가 권한, 장기 유효성, 회수 가능성
3. 무작위, 알파벳, 숫자
4. base64, hex, UUID

**정답: 2.** 시크릿 정의의 본질. 회수 불가능한 것은 시크릿이 아니라 신원입니다.

### Q2. TruffleHog v3에서 `--only-verified`의 의미는?
1. verifier로 살아있다고 확인된 결과만 출력
2. 정규식 매칭만 출력
3. 모든 결과 출력
4. JSON 결과만 출력

**정답: 1.**

### Q3. `Verified: false`의 의미로 가장 정확한 것은?
1. 시크릿이 절대 아님
2. 패턴은 매칭됐으나 verifier 호출 실패 또는 verifier가 없음
3. 도구 오류
4. 정규식 불일치

**정답: 2.**

### Q4. `Raw` 필드를 로그에 그대로 남기지 않는 이유?
1. 디스크 용량 절약
2. 평문 시크릿이 로그에 남아 누출 표면 확대
3. JSON 깨짐
4. 보존 정책 위반

**정답: 2.**

### Q5. TruffleHog와 detect-secrets의 가장 큰 차이?
1. 언어
2. TruffleHog는 라이브 검증(verifier) 지원
3. detect-secrets가 더 빠르다
4. 같다

**정답: 2.**

### Q6. `Keywords()` 사전 필터의 목적?
1. 정확도 향상
2. 비싼 정규식 호출을 피해 성능 확보
3. detector 비활성화
4. 의미 없음

**정답: 2.**

### Q7. Shannon Entropy 단독으로 시크릿을 결정하면?
1. 정확도 100%
2. FP 폭증
3. 빠르다
4. 무관

**정답: 2.**

### Q8. `git rm secret.txt` 후 commit하면 시크릿은?
1. 디스크에서 즉시 삭제
2. unreachable 상태로 남아 GC 전까지 존재
3. fork에도 자동 삭제
4. reflog는 영향 받지 않는다

**정답: 2.** (reflog는 영향 받지만 답은 더 큰 그림.)

### Q9. AWS detector가 key + secret을 함께 매칭하는 이유?
1. 단독 매칭은 FP가 많음
2. AWS 요구 사항
3. 두 값 모두 verifier에 필요
4. 1과 3 모두

**정답: 4.**

### Q10. CI에서 `--no-update`가 권장되는 이유?
