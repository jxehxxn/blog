# 8주차: ITSI와 인프라 모니터링의 심층 이해

보안과 IT 운영은 동전의 양면과 같습니다. 인프라가 흔들리면 보안도 무너집니다. 이번 8주차 수업에서는 Splunk IT Service Intelligence(ITSI)를 써서 서비스 관점에서 시스템을 바라보는 법을 배웁니다. 단순한 로그 수집을 넘어 비즈니스 가치를 지키는 모니터링 체계를 구축해 봅시다.

## 1. 서비스 중심 모니터링으로의 전환

전통적인 모니터링은 "서버가 켜져 있는가?" 또는 "CPU 사용량이 90%를 넘었는가?"와 같은 개별 장비의 상태에 집중했습니다. 하지만 현대의 복잡한 마이크로서비스 아키텍처(MSA)에서는 개별 서버의 상태보다 "사용자가 결제를 할 수 있는가?"라는 질문이 더 중요합니다.

ITSI는 이러한 질문에 답하기 위해 설계되었습니다. 수천 개의 서버와 컨테이너에서 발생하는 데이터를 논리적인 '서비스' 단위로 묶어 관리합니다. 이를 통해 장애가 발생했을 때 어떤 비즈니스 기능이 영향을 받는지 즉시 파악할 수 있습니다. 운영팀은 이제 수만 개의 알람에 시달리는 대신, 비즈니스에 직접적인 타격을 주는 핵심 문제에 집중할 수 있습니다.

## 2. ITSI 아키텍처와 KPI 매핑

ITSI의 핵심은 엔티티(Entity), 서비스(Service), KPI(Key Performance Indicator)의 계층 구조입니다.

### 엔티티와 서비스의 정의
엔티티는 모니터링의 가장 작은 단위입니다. 물리 서버, 가상 머신, 쿠버네티스 포드, 데이터베이스 인스턴스 등이 엔티티가 됩니다. 서비스는 이러한 엔티티들의 논리적 집합입니다. 예를 들어 '결제 서비스'는 웹 서버 포드 10개, Redis 캐시, 오라클 데이터베이스 엔티티로 구성될 수 있습니다. 엔티티는 여러 서비스에 동시에 속할 수 있어 복잡한 의존 관계를 정확히 반영합니다.

### KPI 설계: RED와 USE 방법론
KPI는 서비스의 건강 상태를 측정하는 수치입니다. 효과적인 KPI 선정을 위해 두 가지 방법론을 씁니다.

첫째는 **RED 방법론**입니다. 주로 요청 기반 서비스(API, 웹 서비스)에 씁니다.
- **Rate (요청률)**: 초당 요청 수. 트래픽의 급증이나 급감을 감시합니다.
- **Errors (에러율)**: 실패한 요청 수. HTTP 5xx 에러와 같은 시스템 오류를 추적합니다.
- **Duration (지연 시간)**: 요청 처리에 걸리는 시간. 사용자 경험에 직접적인 영향을 줍니다.

둘째는 **USE 방법론**입니다. 주로 하드웨어 자원(CPU, 메모리, 디스크)에 씁니다.
- **Utilization (사용률)**: 자원이 사용된 시간의 비율. 100%에 도달하면 성능 저하가 시작됩니다.
- **Saturation (포화도)**: 자원이 처리하지 못해 대기 중인 작업의 양. CPU 런큐(Run-queue)나 디스크 대기열이 해당합니다.
- **Errors (에러)**: 자원 수준에서 발생한 에러 횟수. 디스크 I/O 에러나 네트워크 패킷 드롭을 감시합니다.

ITSI는 이 지표들을 조합해 0점에서 100점 사이의 서비스 점수를 산출합니다.

### 임계치 설정과 적응형 임계치
임계치 설정은 ITSI의 강력한 기능 중 하나입니다. 고정된 수치를 입력하는 정적 임계치 외에도, 머신러닝을 사용한 적응형 임계치(Adaptive Thresholding)를 지원합니다. 과거 데이터를 학습해 요일별, 시간별 정상 범위를 스스로 계산합니다. 월요일 아침의 트래픽 급증은 정상으로 간주하고, 일요일 새벽의 갑작스러운 트래픽은 이상 징후로 판단하는 식입니다. 이를 통해 오탐(False Positive)을 획기적으로 줄일 수 있습니다.

### 서비스 분석기와 글래스 테이블
서비스 분석기(Service Analyzer)는 전체 서비스의 의존 관계를 한눈에 보여주는 대시보드입니다. 상위 서비스의 점수가 떨어지면 하위 어떤 서비스에서 문제가 시작되었는지 추적할 수 있습니다.

글래스 테이블(Glass Tables)은 이 데이터를 시각화하는 도구입니다. 복잡한 인프라 구조를 직관적인 다이어그램으로 그려낼 수 있습니다. 경영진에게는 서비스 가용성을 보여주고 운영팀에게는 장애 지점을 즉시 노출합니다.

## 3. 쿠버네티스 로깅 아키텍처 심층 분석

쿠버네티스 환경에서 로그가 생성되어 Splunk로 전달되는 과정은 복잡합니다. 이를 이해해야 정확한 모니터링이 가능합니다.

### 로그 생성: stdout과 stderr의 흐름
쿠버네티스에서 실행되는 컨테이너는 로그를 파일에 직접 쓰지 않는 것이 원칙입니다. 대신 표준 출력(stdout)과 표준 에러(stderr)로 로그를 내보냅니다. 컨테이너 런타임(Docker 또는 containerd)은 이 스트림을 가로채서 노드의 특정 경로(`/var/log/pods/` 또는 `/var/log/containers/`)에 JSON 형식의 파일로 저장합니다.

이 과정에서 런타임은 로그에 타임스탬프와 스트림 종류(stdout/stderr) 정보를 덧붙입니다. 노드 수준의 로그 파일은 용량이 커지면 순환(Rotation)되도록 설정되어 있습니다. 로그 파일이 너무 빨리 쌓여서 수집되기 전에 지워지지 않도록 주의해야 합니다.

### 로그 수집: 데몬셋 패턴
노드에 저장된 로그 파일을 수집하기 위해 주로 데몬셋(DaemonSet) 패턴을 사용합니다. 모든 노드에 하나씩 수집 에이전트를 띄우는 방식입니다. Fluentd, Fluent Bit 또는 Splunk OpenTelemetry Collector가 이 역할을 수행합니다.

에이전트는 노드의 로그 파일을 읽어 들여 메타데이터를 추가합니다. 어떤 네임스페이스에서 발생했는지, 포드 이름은 무엇인지, 어떤 라벨이 붙어 있는지 등의 정보를 로그에 붙여 Splunk로 전송합니다. 이 메타데이터가 있어야 나중에 Splunk에서 특정 서비스의 로그만 골라낼 수 있습니다.

### 사이드카 패턴의 활용
만약 애플리케이션이 stdout으로 로그를 내보낼 수 없는 구조라면 사이드카(Sidecar) 패턴을 씁니다. 애플리케이션 컨테이너 옆에 로그 수집용 컨테이너를 함께 띄워 로그 파일을 읽어내는 방식입니다. 이 방식은 자원을 더 많이 쓰지만, 레거시 애플리케이션을 컨테이너화할 때 유용합니다.

### Splunk OpenTelemetry Collector의 역할
최근에는 OpenTelemetry(OTel)가 표준으로 자리 잡고 있습니다. Splunk OTel Collector는 로그뿐만 아니라 메트릭과 트레이스 데이터를 동시에 수집합니다. 이를 통해 로그와 성능 지표를 연결해 분석할 수 있는 강력한 가시성을 제공합니다. OTel Collector는 수집한 데이터를 가공(Processor)하고, 여러 목적지로 전송(Exporter)하는 유연한 구조를 가집니다. 특히 쿠버네티스 API 서버와 통신하여 실시간으로 인프라 변화를 감지하고 메타데이터를 동기화합니다.

## 4. AWS 인프라 모니터링 아키텍처

AWS 환경은 서비스마다 로그 생성 방식이 다릅니다. Splunk는 이를 통합하기 위해 다양한 수집 경로를 제공합니다.

### 주요 로그 소스
- **CloudWatch Logs**: EC2, Lambda, RDS 등 대부분의 AWS 서비스가 로그를 저장하는 곳입니다.
- **VPC Flow Logs**: 네트워크 트래픽의 흐름을 기록합니다. 보안 분석의 핵심 데이터입니다.
- **CloudTrail**: AWS 계정 내에서 발생하는 모든 API 호출을 기록합니다. 누가 어떤 자원을 수정했는지 추적할 때 씁니다.
- **S3 Access Logs**: 데이터 저장소에 대한 접근 기록을 남깁니다.

### 데이터 전송 경로: Firehose vs Add-on
가장 권장되는 방식은 **Kinesis Data Firehose**를 사용하는 것입니다. CloudWatch Logs에서 발생한 데이터를 Firehose로 보내고, Firehose가 이를 Splunk의 HTTP Event Collector(HEC)로 직접 밀어넣습니다. 이 방식은 서버리스 구조라 관리가 편하고 대량의 데이터를 실시간으로 처리하기에 적합합니다. 중간에 데이터가 유실되지 않도록 S3 백업 설정도 가능합니다.

전통적인 방식으로는 **Splunk Add-on for AWS**를 사용합니다. SQS와 S3를 조합해 Splunk가 데이터를 주기적으로 가져오는(Pull) 방식입니다. 설정이 직관적이라 초기 구축에 유리하지만, 데이터 양이 많아지면 지연 시간이 발생할 수 있습니다. 대규모 환경에서는 Firehose를 통한 Push 방식을 우선적으로 고려해야 합니다.

## 5. 메트릭과 이벤트: stats와 mstats의 기술적 차이

Splunk에서 데이터를 다루는 방식은 크게 이벤트(Events)와 메트릭(Metrics)으로 나뉩니다.

### 이벤트와 stats
이벤트는 원본 로그 그 자체입니다. 텍스트 검색이 가능하고 유연하지만, 데이터 양이 많아지면 집계 속도가 느려집니다. `stats` 명령어는 이 이벤트 데이터를 검색 시점에 계산(Schema-on-read)하여 결과를 보여줍니다. 인덱서가 원본 데이터를 하나씩 읽어야 하므로 자원을 많이 씁니다.

### 메트릭과 mstats
메트릭은 수치화된 데이터입니다. CPU 사용률 85%와 같은 데이터를 미리 정의된 형식으로 저장합니다. Splunk의 메트릭 스토어는 시계열 데이터 처리에 최적화된 특수 인덱스를 사용합니다.

`mstats` 명령어는 이 메트릭 인덱스에서 작동합니다. 수백만 개의 데이터 포인트를 순식간에 처리할 수 있어 대규모 인프라의 실시간 대시보드를 구성할 때 필수적입니다. 인프라 모니터링에서는 로그(이벤트)로 원인을 파악하고, 메트릭으로 현상을 빠르게 감시하는 전략이 필요합니다.

## 6. 실전 가이드: ITSI 서비스 맵 구축하기

성공적인 ITSI 구축을 위해서는 다음 단계를 따릅니다.

1. **서비스 분해**: 비즈니스 프로세스를 기술적인 서비스 단위로 나눕니다.
2. **엔티티 정의**: 각 서비스에 속한 자원들을 식별하고 라벨링합니다.
3. **KPI 선정**: 서비스의 건강을 대변할 수 있는 핵심 지표를 고릅니다. (RED 패턴 권장)
4. **의존성 연결**: 서비스 간의 관계를 설정해 장애 전파 경로를 시각화합니다.
5. **임계치 최적화**: 초기에는 정적 임계치를 쓰고, 데이터가 쌓이면 적응형 임계치로 전환합니다.
6. **알림 정책 설정**: Notable Event Aggregation Policies(NEAP)를 써서 중복 알림을 줄이고 의미 있는 경고만 남깁니다.

## 7. AIOps와 예측 분석

ITSI는 단순한 모니터링을 넘어 인공지능을 활용한 운영(AIOps)을 지향합니다.

### 이상 징후 탐지 (Anomaly Detection)
KPI 데이터의 패턴을 분석하여 평소와 다른 움직임을 포착합니다. 갑작스러운 트래픽 증가나 응답 시간 지연이 발생하면 시스템이 자동으로 경고를 보냅니다. 이는 사람이 일일이 임계치를 설정하기 어려운 수천 개의 지표를 관리할 때 매우 유용합니다.

### 장애 예측 (Predictive Analytics)
과거의 장애 패턴을 학습하여 미래에 발생할 장애를 미리 예측합니다. "현재 추세라면 30분 뒤에 결제 서비스의 점수가 40점 이하로 떨어질 확률이 80%입니다"와 같은 정보를 제공합니다. 이를 통해 장애가 발생하기 전에 선제적으로 조치할 수 있습니다.

## 결론: 장애를 예측하는 방패

인프라 모니터링은 단순히 장애를 기록하는 일기가 아닙니다. ITSI와 같은 도구를 써서 서비스 관점의 가시성을 확보하면 장애를 미리 예측하고 대응할 수 있는 방패가 됩니다. 이번 주 배운 내용을 바탕으로 여러분의 환경에 맞는 서비스 맵을 그려보시기 바랍니다. 궁금한 점은 언제든 질문해 주세요.
