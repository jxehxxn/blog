빅테크 보안 현장에서 수조 개의 이벤트를 다루다 보면 쿼리 하나가 전체 클러스터 성능을 좌우하는 걸 자주 봅니다.
단순히 결과를 뽑는 걸 넘어 성능까지 고려한 쿼리 작성이 전문가의 실력을 가릅니다.
이번 주에는 join, 서브서치, 데이터 모델, transaction 명령어를 깊게 파헤쳐 보겠습니다.

### 1. join과 서브서치의 함정
많은 입문자가 SQL 습관 때문에 join을 먼저 찾습니다.
하지만 스플렁크에서 join은 성능의 적입니다.
메모리 제한이 있고 대규모 데이터셋에서는 실행 속도가 급격히 떨어집니다.
서브서치([search ...])도 마찬가지입니다.
결과가 1만 건을 넘어가면 제대로 작동하지 않거나 쿼리가 중단될 수 있습니다.

### 2. transaction vs stats
transaction 명령어는 이벤트 간의 관계를 묶어주기에 편리합니다.
로그인부터 로그아웃까지의 흐름을 한눈에 보여주죠.
문제는 이 명령어가 모든 데이터를 메모리에 올린 뒤 계산한다는 점입니다.
수백만 건의 세션을 처리할 때는 stats를 쓰는 게 훨씬 빠릅니다.
stats count, min(_time) as start, max(_time) as end by session_id 같은 방식이 서버 부하를 줄이는 정석입니다.

### 3. 데이터 모델(Data Models)의 힘
반복되는 복잡한 쿼리는 데이터 모델로 정의해두면 좋습니다.
가속화(Acceleration) 기능을 켜면 검색 속도가 수십 배 빨라집니다.
보안 분석가라면 CIM(Common Information Model)을 활용해 장비 제조사에 상관없이 일관된 분석 환경을 구축해야 합니다.

### 4. 빅테크 규모에서의 쿼리 최적화
실제 운영 환경에서는 join 대신 stats나 lookup을 조합합니다.
두 인덱스의 데이터를 합쳐야 한다면 (index=A OR index=B)로 한 번에 불러온 뒤 stats로 묶으세요.
이게 스플렁크 분산 검색 아키텍처를 가장 잘 활용하는 방법입니다.
불필요한 필드는 fields 명령어로 미리 제거하는 습관도 중요합니다.

### 자가 진단 퀴즈
1. 다음 중 대규모 데이터셋에서 가장 권장되지 않는 명령어는?
   A) stats
   B) join
   C) eval
   D) lookup

2. transaction 명령어 대신 stats를 권장하는 주된 이유는?
   A) 문법이 더 쉬워서
   B) 시각화가 잘 돼서
   C) 메모리 효율과 검색 속도 때문
   D) 필드 추출이 자동이라서

3. 데이터 모델 가속화를 사용하면 얻는 이점은?
   A) 라이선스 비용 절감
   B) 검색 속도 향상
   C) 로그 저장 용량 감소
   D) 대시보드 디자인 개선

**정답:** 1-B, 2-C, 3-B
